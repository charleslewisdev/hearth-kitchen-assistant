| 10 | 2026-06-17 | **Canonical Food entity, auto-mapped from free-text** (auto-create + mergeable; carries department, default unit, per-food settings; seeded food→department dictionary; deterministic fuzzy matching) | Gives the full structured payoff while feeling like free-text typing — no upfront catalog management, fits non-technical-first. | 11 | 2026-06-17 | **Comprehensive automated testing is first-class; the deterministic core is built test-first (TDD)** — scaling, unit conversion, shopping aggregation, food-mapping, parsers, receipt itemization all have thorough unit-test suites; CI gates on green | These are pure rule-driven behaviors where silent wrongness destroys user trust ("my shopping list is wrong"). | 12 | 2026-06-17 | **Per-ingredient scaling = `scalingMode` enum** (`linear` default / `fixed` / `sublinear` / `manual`) with **smart category defaults** from the seeded Food dictionary; **Food suggests, RecipeIngredient stores the resolved rule** (context wins) | Delivers accurate scaling no FOSS competitor has, invisible to non-technical users (good defaults) yet controllable; context-dependent (pan-oil vs dressing-oil) needs per-recipe resolution. Implemented as a pure, exhaustively-tested `scale()` function. | — |
| 13 | 2026-06-17 | **Units: canonical base (g/ml/count) stored; exact in-family conversion + aggregation; cross-family (volume↔weight) deferred behind optional per-Food `density`** (unknown density ⇒ list separately, never guess); unitless ("to taste") first-class non-aggregating. **Display defaults to US customary, per-household metric toggle.** | Covers ~95% of shopping-list math deterministically without heavy density seed data; density hook allows later upgrade with no redesign; US default fits the primary audience while staying storage-agnostic. | — |
| 14 | 2026-06-17 | **Deterministic-vs-LLM boundary: core is 100% deterministic forever** (planning, lists, aggregation, scaling, units, search, sharing, budget math). **Import & receipt parsing have a deterministic baseline** (JSON-LD scrape, **tesseract.js local OCR**, PDF/text heuristics) **+ an optional BYO-key LLM/vision provider** for messy/handwritten inputs. **Image import works in-the-box with no API key.** | Directly answers "does this need an LLM?" — no for the core, optional-only for two fuzzy jobs. Beats Mealie (which now requires OpenAI for image import). Upholds deterministic-first + in-the-box self-host. | — |
| 15 | 2026-06-17 | **Leftovers/freezer/freshness unified as a "portion ledger"**: cooking produces portions → allocated to meal / leftover-lunch / frozen `StoredMeal`; planning can fill a slot from fresh cook, leftover, or freezer; per-Food `shelfLifeDays` hints drive deterministic freshness advisories (Grocy Due-Score idea applied to cook order). Build is staged (lightweight slice first) but the concept is locked now. | One coherent model delivers four unserved differentiators (C2/C3/C4/D3); locking the concept early keeps the data model from fighting it later. | — |
| 16 | 2026-06-17 | **Search = Postgres full-text** (no external search service). **MVP filters: ingredient, meal-of-day, tag/category** (free-text search included); total-time/cuisine/dietary/source/"cookable-now" deferred to Core. | Avoids an extra service (in-the-box, non-technical-friendly); Postgres FTS is sufficient at household scale; lean MVP filter set leverages the Food model. | — |
| 17 | 2026-06-17 | **Shopping list: perimeter/interior department default + per-household drag-to-reorder**; Foods map to departments (seeded + override); **check-off state persisted server-side and synced across devices** (Keep-style); active list = plan-generated + free-form adds; multiple named store layouts deferred. | Ships the user's described default out of the box plus cheap high-delight store-order customization; synced check-off enables shared live shopping. | — |
| 18 | 2026-06-17 | **Sale ads are tiered: T1 = structured sale-list import (CSV/JSON) + manual** (deterministic, community-shareable), **T2 = flyer photo/PDF OCR/LLM**, **T3 = aggregator API/scraping (feasibility + legal/ToS gated)**. Sale items resolve to canonical Foods; payoff features (sale-aligned recipe surfacing, stock-up/freeze flags) are deterministic once data exists. Food model is sale-aware from the start. | Reuses the Food entity for matching (Decision 10 payoff); deterministic entry tier with no legal risk; defers risky scraping behind feasibility gates. | — |
| 19 | 2026-06-17 | **Budget = full loop (north-star), built in two steps.** Step 1: receipts (photo/PDF) → OCR/LLM parse → line items → **learned per-store Food alias mappings** → Purchase records → spend analytics (time/category/Food/store). Step 2: **price-aware planning** — estimate plan cost beforehand, estimated-vs-actual variance, cheaper-swap nudges. Foods carry observed prices from receipts + sales. | The literal "save money over time" payoff (I4) and product apex; reuses the Food entity a third time; two-step build de-risks the deepest feature. | — |
| 20 | 2026-06-17 | **Pantry/inventory = lightweight, opt-in** (Foods + rough quantities; powers "cookable now" + shopping de-dup); **full Grocy-style stock-keeping (barcode/exact/expiry) is a far-later power-user layer**. | Adds real waste/over-buy reduction and feeds price-aware planning without taxing non-technical users; reuses the Food model again. | — |
